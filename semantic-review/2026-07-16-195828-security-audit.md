# Audit Keamanan & Gaps: WaKasir SaaS

Codebase WaKasir adalah WhatsApp Commerce Bot SaaS multi-tenant — Laravel 12 backend, Angular 22 frontend, credential sensitif dari tiga pihak ketiga (Meta, Midtrans, RajaOngkir) disimpan per-bisnis di database. `wa_access_token` sudah dienkripsi dengan benar, isolasi tenant via `business_id` konsisten, dan `$hidden` di model Business memblokir kebocoran token ke API response. Tapi ada sepuluh gap yang perlu ditangani sebelum production.

**Watch for (confirmed):** CORS `allowed_origins: ['*']` + `supports_credentials: true` belum dikunci ke domain production. Midtrans server key dan RajaOngkir API key tersimpan plain varchar — spec menyebut enkripsi tapi migrasi tidak mengimplementasikannya. Subscription quota dicatat *sesudah* bot merespons, bukan dicek *sebelumnya*, sehingga bisnis dengan kuota habis tetap dilayani penuh.

---

## High-level view

CORS dikonfigurasi dengan kombinasi `allowed_origins: ['*']` dan `supports_credentials: true` yang dilarang spesifikasi. Browser modern memblokir respons semacam ini, tapi masalah inti adalah origin belum pernah dikunci ke domain production.

Tiga API key — Midtrans server key, Midtrans client key, RajaOngkir API key — disimpan plain varchar di tabel `businesses`. Spec eksplisit menyebut enkripsi untuk ketiganya, tapi migrasi dan `BusinessService::updateIntegration()` tidak mengimplementasikannya. Berbeda dengan `wa_access_token` yang sudah benar.

`PaymentService.handleWebhook()` mengakses `$webhookData['signature_key']` dan `$webhookData['status_code']` tanpa null guard, dan mencari order *sebelum* verifikasi selesai. Jika order tidak ditemukan, verifikasi menggunakan global server key (fallback `$this->midtransService`), bukan key merchant yang seharusnya — membuka potensi bypass signature dengan pengetahuan global key.

`authGuard` di Angular hanya memeriksa keberadaan token di localStorage. Token yang sudah dicabut server tetap meloloskan akses ke semua halaman sampai request pertama menghasilkan 401.

Subscription quota direkam di langkah 5 dari `ProcessIncomingWhatsAppMessage::handle()`, setelah bot memproses, pesan dikirim, dan order dibuat. Kuota habis tidak menghentikan pemrosesan.

`connectWhatsApp` tanpa rate limiting berarti pengguna terautentikasi bisa men-spam dua outbound request ke Meta API per panggilan, yang bisa memicu rate limiting atau suspension Meta App yang dipakai semua tenant.

Pesan teks dari pelanggan WhatsApp masuk tanpa batas panjang — langsung ke `message_logs.content` (TEXT) dan bisa sampai ke `addresses.full_address` (varchar 255).

`DatabaseSeeder` menggunakan `PRAGMA foreign_keys = OFF` yang hanya valid di SQLite; di MySQL ini silent no-op, plus seeder mengandung kredensial hardcoded tanpa guard `APP_ENV`.

`OrderController.updateStatus` menerima transisi status apapun; `StateMachine::canTransition()` sudah ada di kodebase tapi tidak digunakan.

`check_order` intent di IDLE meminta nomor order tapi tidak ada state atau handler yang memproses reply tersebut — pelanggan masuk ke fallback.

1. **CORS wildcard + credentials** — `allowed_origins: ['*']` dengan `supports_credentials: true` dilarang spesifikasi CORS. Ubah ke domain eksplisit via `env('CORS_ALLOWED_ORIGINS')` di `cors.php`.

3. **PaymentService.handleWebhook: null access + urutan operasi** — `$webhookData['status_code']` dan `$webhookData['signature_key']` diakses tanpa guard; order dicari sebelum verifikasi; jika order null, verifikasi menggunakan global key. Tambahkan null guard di awal, pindahkan pencarian order ke setelah verifikasi berhasil.

4. **authGuard tidak validasi server** — Token dicabut atau expired lolos ke semua halaman. Tambahkan resolving `/me` di guard atau gunakan `CanActivate` berbasis observable yang memanggil server sebelum render.

5. **Subscription quota dicek setelah respons** — `recordUsage()` dipanggil di langkah 5 setelah bot merespons dan pesan terkirim. Pindahkan `checkQuota()` ke sebelum `$messageHandler->handle()` dan kembalikan pesan "kuota habis" jika exceeded.

6. **connectWhatsApp tanpa rate limiting** — Endpoint melakukan dua outbound ke Meta API per panggilan tanpa throttle tambahan. Tambahkan `throttle:5,1` per user.

7. **Input pesan tanpa length limit** — Teks bebas dari pelanggan masuk tanpa sanitasi ke `message_logs` dan `addresses.full_address`. Tambahkan `mb_substr($text, 0, 4096)` di `ProcessIncomingWhatsAppMessage` sebelum meneruskan ke handler.

8. **DatabaseSeeder PRAGMA hanya SQLite** — `PRAGMA foreign_keys = OFF` no-op di MySQL. Bungkus dengan `if (DB::getDriverName() === 'sqlite')`. Tambahkan guard `APP_ENV !== 'production'` untuk mencegah penghapusan data production.

9. **updateStatus tanpa validasi state machine** — `StateMachine::canTransition()` ada tapi tidak dipakai di `OrderController`. Gunakan untuk validasi sebelum update.

10. **check_order dead end** — `handleIdle()` meminta nomor order tapi state tetap `IDLE`; reply nomor order tidak diproses. Implementasikan state `CHECKING_ORDER` atau tambahkan pencocokan pola `ORD-\d{8}-\d{4}` di IDLE.

## CORS: Wildcard Origin + Credentials

```php
// config/cors.php
'allowed_origins'      => ['*'],
'supports_credentials' => true,
```

Spesifikasi CORS melarang kombinasi ini secara eksplisit — browser akan memblokir respons, bukan mengirimnya. Di production Laravel + Sanctum SPA, `allowed_origins` harus berisi domain frontend secara eksplisit. Perbaikan:

```php
'allowed_origins'      => explode(',', env('CORS_ALLOWED_ORIGINS', 'http://localhost:4200')),
'supports_credentials' => true,
```

Dengan `CORS_ALLOWED_ORIGINS=https://app.wakasir.com` di `.env` production. Tambahkan juga ke `.env.example`.

## API Key Midtrans & RajaOngkir Plain-Text

`prompt.md` Bagian 8 menyebutkan:

> `midtrans_server_key (encrypted), midtrans_client_key (encrypted), rajaongkir_api_key (encrypted)`

Migrasi `create_businesses_table.php` mendefinisikan ketiganya sebagai `string` (varchar 255) biasa. `BusinessService::updateIntegration()` menyimpannya langsung tanpa `encrypt()`. Jika database bocor — melalui backup yang tidak terenkripsi, SQL injection, atau akses staff — Midtrans server key seluruh tenant terbaca plain. Midtrans server key memberi akses penuh ke semua transaksi merchant: refund, cancel, status check.

Kolom perlu diubah ke `text` (ciphertext lebih panjang dari 255 karakter), disimpan dengan `encrypt()`, dan dibaca dengan `decrypt()` on-the-fly — persis pola yang sudah ada di `getWaAccessTokenDecrypted()`.

## PaymentService.handleWebhook: Urutan Operasi Berbahaya

```php
$signatureKey = $webhookData['signature_key'];   // tidak ada null check

$order    = Order::where('order_number', $orderId)->first();
$business = $order?->business;

// Jika order null, $business null → fallback ke $this->midtransService (global key)
$midtrans = ($business && $business->midtrans_server_key)
    ? new MidtransService(serverKey: $business->midtrans_server_key, ...)
    : $this->midtransService;   // ← global key WaKasir digunakan untuk verifikasi

if (!$midtrans->verifyWebhookSignature(
    $orderId,
    $webhookData['status_code'],  // tidak ada null check
    $webhookData['gross_amount'],
    $signatureKey
)) { ... }
```

Masalah pertama: akses array tanpa guard. Payload malformed akan melempar `Undefined array key` yang ditangkap job sebagai error dan di-retry — bukan ditolak secara eksplisit.

Masalah kedua: ketika `order_id` tidak valid, `$business` null, dan verifikasi berjalan menggunakan global server key WaKasir. Seseorang yang mengetahui global key bisa membuat payload dengan `order_id` sembarang, hitung signature terhadap global key, dan lolos verifikasi — meskipun tidak ada order yang cocok (`$order` null menyebabkan `return false` berikutnya, tapi jalur ini masih dieksekusi).

Urutan yang benar: (1) validasi semua field wajib ada, (2) cari order, (3) gunakan key merchant untuk verifikasi, (4) proses.

## authGuard: Client-Side Only

```typescript
export const authGuard: CanActivateFn = () => {
  const token = localStorage.getItem('token');
  if (token) return true;
  return router.createUrlTree(['/login']);
};
```

Token dicabut via `POST /logout` di tab lain, token expired secara Sanctum, atau token tersisa setelah manipulasi manual — semuanya lolos. `authInterceptor` menangani 401 dengan benar, tapi reactive: pengguna sudah di halaman dengan data sensitif (omzet, data pelanggan) sebelum request pertama dikirim.

Opsi terbaik untuk Angular 22: gunakan `CanActivateFn` yang mengembalikan `Observable<boolean>` dengan call ke `AuthService.me()`, dengan cache singkat untuk menghindari request di setiap navigasi.

## Subscription Quota Dicatat Setelah Respons

```php
// ProcessIncomingWhatsAppMessage::handle()
$response = $messageHandler->handle(...);          // step 2: bot memproses + order dibuat
dispatch(new SendWhatsAppNotification(...));        // step 3: pesan terkirim ke pelanggan
$subscriptionService->recordUsage($businessId);    // step 5: baru dicek
```

`recordUsage()` mengembalikan `false` jika kuota habis, tapi tidak ada yang membaca return value ini. Bot tetap memproses, QR tetap digenerate, pesan tetap dikirim. Perbaikan: panggil `checkQuota()` sebelum `$messageHandler->handle()`:

```php
$quota = $subscriptionService->checkQuota($this->businessId);
if ($quota['is_exceeded']) {
    dispatch(new SendWhatsAppNotification($this->waNumber,
        'Maaf, layanan bot sedang tidak tersedia. Silakan hubungi admin toko.',
        $this->businessId));
    return;
}
```

## connectWhatsApp: Tidak Ada Rate Limiting

`POST /api/business/whatsapp/connect` diproteksi `auth:sanctum` tapi tidak punya throttle tambahan. `WhatsAppService::exchangeCodeForToken()` melakukan dua outbound ke Meta: `graph.facebook.com/oauth/access_token` dan `graph.facebook.com/me/businesses`. Pengguna yang token-nya masih valid bisa memicu puluhan atau ratusan request ke Meta API dalam menit, memicu rate limit atau suspension Meta App yang dipakai oleh seluruh tenant WaKasir.

---

## Input Pesan Tanpa Length Limit

`ProcessIncomingWhatsAppMessage` meneruskan `$this->message` langsung ke dua path:

1. `MessageLog::create(['content' => $this->message])` — kolom TEXT, tidak ada batasan di level aplikasi
2. `MessageHandler::handle()` → `handlePaidAwaitingAddress()` → `Address::create(['full_address' => $message])` — kolom `varchar(255)` di migrasi, tidak ada validasi panjang

Meta membatasi `text.body` ke 4096 karakter, tapi request yang langsung dikirim ke webhook endpoint bisa mengandung payload lebih panjang. Truncation varchar tanpa error handling bisa menghasilkan alamat pengiriman yang terpotong tanpa notifikasi.

---

## DatabaseSeeder: PRAGMA + Kredensial Hardcoded

`PRAGMA foreign_keys = OFF` adalah SQLite-only. Di MySQL production, driver mengembalikan error atau silent no-op — tidak ada yang menonaktifkan foreign key check yang seharusnya melindungi integritas saat truncate.

Yang lebih berbahaya: seeder tidak ada perlindungan environment. `php artisan db:seed` di production akan menghapus semua data user, order, produk, dan membuat akun `admin@wakasir.dev` / `password`. Tambahkan di awal `run()`:

```php
if (app()->isProduction()) {
    $this->command->error('Seeder tidak boleh dijalankan di production!');
    return;
}
```

## OrderController.updateStatus: State Machine Tidak Dipakai

`StateMachine.php` mendefinisikan `$transitions` yang lengkap dan `canTransition()` yang sudah bisa dipakai. `OrderController::updateStatus()` hanya memvalidasi nilai string status, tidak transisinya. Dashboard bisa set order dari `pending` (belum bayar, belum ada Payment record) langsung ke `completed`, yang akan masuk ke laporan revenue dan memicu `completed_at` timestamp.

## check_order: Dead End di IDLE

```php
if ($intent === 'check_order') {
    return [
        'text'  => "Silakan kirim nomor order Anda ...",
        'state' => StateMachine::STATES['IDLE'],  // state tidak berubah
    ];
}
```

State tetap `IDLE`. Ketika pelanggan membalas dengan nomor order, `handleIdle()` dipanggil dengan string `ORD-20260716-0001` — tidak ada intent yang cocok, bot menampilkan menu utama. Dua kali tidak cocok → `FALLBACK_CS`.

Untuk fix minimal tanpa state baru: di `handleIdle()`, tambahkan pencocokan pola regex sebelum blok intent lain:

```php
if (preg_match('/ORD-\d{8}-\d{4}/i', $message, $matches)) {
    $order = Order::where('order_number', $matches[0])
        ->where('business_id', $business->id)
        ->with('payment')
        ->first();
    // return status order
}
```

## .env.example: Variabel Kritikal Tidak Ada

`.env.example` tidak mengandung satu pun dari variabel yang dibutuhkan `config/services.php`:

```
# WAJIB diisi sebelum deployment
MIDTRANS_SERVER_KEY=
MIDTRANS_CLIENT_KEY=
MIDTRANS_IS_PRODUCTION=false

RAJAONGKIR_API_KEY=
RAJAONGKIR_BASE_URL=https://api.rajaongkir.com/starter

WA_PHONE_ID=
WA_ACCESS_TOKEN=
WA_WEBHOOK_VERIFY_TOKEN=
META_APP_ID=
META_APP_SECRET=
META_EMBEDDED_SIGNUP_CONFIG_ID=

CORS_ALLOWED_ORIGINS=http://localhost:4200
```

`config/services.php` akan mengembalikan `null` untuk semua key ini, Midtrans akan melempar error saat pertama kali dipakai, dan webhook Meta akan gagal challenge verification karena `WA_WEBHOOK_VERIFY_TOKEN` default ke string `'wa_verify_default'` yang tidak aman.
